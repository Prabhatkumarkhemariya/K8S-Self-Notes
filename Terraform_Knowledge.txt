Terraform – Description
Terraform is an open-source Infrastructure as Code (IaC) tool that allows you to define, provision, and manage infrastructure using declarative configuration files. It is used to automate infrastructure provisioning, ensure consistency across environments, enable version control of infrastructure, and support scalable, repeatable deployments across multiple cloud and on-prem platforms.

Terraform is an Infrastructure as Code (IaC) tool developed by HashiCorp that enables DevOps engineers to provision, manage, and version cloud infrastructure using declarative configuration files. It allows infrastructure to be defined as code, making deployments repeatable, consistent, and automated across environments.

Terraform supports AWS, Azure, GCP, and on-premises platforms, enabling the creation and management of resources such as VPCs, EC2 instances, EKS clusters, IAM roles, security groups, and load balancers. It uses a state file to track infrastructure changes and provides features like execution plans, modular design, and resource dependency management to safely apply updates.

By integrating Terraform with CI/CD pipelines, teams can achieve automated infrastructure provisioning, version control, scalability, and reduced manual errors, making it a core tool in modern DevOps and cloud-native environments.

4️ State File (terraform.tfstate)
Terraform state file is a JSON document that stores metadata, outputs, resource mappings, attributes, and dependency information required to manage infrastructure lifecycl

Metadata in the Terraform state file includes resource IDs, attributes, dependencies, and provider details that allow Terraform to manage infrastructure lifecycle accurately.

terraform.tfstate
├── version   (version represents the STATE FILE FORMAT VERSION)
├── terraform_version (  terraform_version tells which Terraform binary version
last created or modified the state file)
├── serial
├── lineage
├── outputs
└── resources
    └── instances
        └── attributes

Stores current infrastructure state
Maps resources ↔ real cloud objects
Used for drift detection
📌 Stored:
Locally (default)
Remotely (S3 + DynamoDB – recommended)

5️⃣ Backend
Terraform backend is responsible for storing the state file remotely, enabling state locking, collaboration, security, and consistency across teams.

Terraform stores state locally on your system.
But in team / production environments, this is not safe or scalable.
So we use a backend to store the state remotely and securely.

Terraform Backend Workflow
terraform init
     ↓
Configure backend
     ↓
State stored remotely
     ↓
terraform plan
terraform apply


Backend defines where and how state is stored.
terraform {
  backend "s3" {
    bucket         = "my-terraform-state-bucket"
    key            = "dev/vpc/terraform.tfstate"
    region         = "ap-south-1"
    dynamodb_table = "terraform-lock"
    encrypt        = true
  }
}

What each field means:
bucket → S3 bucket for state
key → Path of state file
region → AWS region
dynamodb_table → State locking
encrypt → Encrypt state file

usess
✔ Enables:
Remote state
State locking
Team collaboration

6️⃣ Dependency Graph
Terraform Dependency Graph is an internal mechanism that automatically determines the correct order of resource creation and destruction based on dependencies, without requiring manual intervention.

Terraform automatically determines:
Resource creation order
Resource destruction order
Creation Order:
Subnet (network layer)
EC2 (compute inside subnet)
ALB (routes traffic to EC2)
Destruction Order:

ALB

EC2

Subnet
Terraform handles this automatically.
Example:
subnet → ec2 → alb
No need to manually manage dependencies.
3. No Manual Ordering Needed
You don’t need to write scripts like:
create subnet
wait
create ec2
wait
create alb
Terraform handles everything.
####
Terraform Commands Related to Dependency Graph
View Dependency Graph:
terraform graph
You can visualize it using Graphviz:
terraform graph | dot -Tpng > graph.png

##############################
main.tf 
Definition
main.tf is the primary Terraform configuration file where you define the main infrastructure resources that Terraform will create and manage.
main.tf is the primary Terraform configuration file that defines the core infrastructure resources and module calls for a project.

In simple words:
main.tf contains the actual infrastructure code such as EC2, VPC, Subnet, ALB, S3, etc.
1️⃣ main.tf
Contains:
Providers
Resources
Data sources

✅ 1. Provider Usage (Sometimes)
Although providers are often placed in provider.tf, they can exist in main.tf.
✅ 2. Resource Definitions
resource "aws_instance" "web" {
  ami           = "ami-123"
  instance_type = "t2.micro"
}
✅ 3. Module Calls
module "vpc" {
  source = "./modules/vpc"
}

✅ 4. Data Sources (Optional)
data "aws_ami" "amazon_linux" {
  most_recent = true
}

Why is main.tf Called a Core Terraform File?
Terraform projects usually contain multiple .tf files, but:
main.tf holds the core logic
It is the entry point for infrastructure definition
Terraform reads all .tf files together, but main.tf is the standard naming convention
📌 The name is not mandatory, but industry best practice.

2️⃣ variables.tf

6️⃣ backend.tf
backend.tf is a Terraform configuration file that defines where Terraform stores the state file (terraform.tfstate) remotely instead of on a local machine.
🔹 Example (S3 Backend)
terraform {
  backend "s3" {
    bucket         = "terraform-state-bucket"
    key            = "dev/terraform.tfstate"
    region         = "ap-south-1"
    dynamodb_table = "terraform-lock"
    encrypt        = true
  }
}

bucket → name of the S3 bucket
key → path + filename inside the S3 bucket
Folder: dev
File: terraform.tfstate
Helps separate environments:
##region = "ap-south-1"

##dynamodb_table = "terraform-lock"
dynamodb_table → name of DynamoDB table
Used for state locking
Prevents:
Two users running terraform apply at the same time
State file corruption
📌 Table must exist with:
Primary key: LockID
👉 Remote state helps with:
Team collaboration
State locking
Backup & security
#########
encrypt = true
Enables server-side encryption for the state file in S3
Terraform state may contain:
Passwords
Secrets
Private IPs
📌 true = encrypted
📌 false = not secure (not recommended)

️⃣ versions.tf
📌 Definition
versions.tf enforces Terraform and provider version constraints.
Example
terraform {
  required_version = ">= 1.5.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}
versions.tf ensures consistent Terraform and provider versions across environments.

Terraform Module
A module is a reusable block of Terraform code that helps standardize and scale infrastructure.	
A module is a reusable, configurable collection of Terraform files.
📌 Root module = current directory
📌 Child module = called by root module

Why Use Modules?
Reusability
Clean code
Standardization
Scalability
Team collaboration
##########
Module Types
Type	Description
Root module 	Main working directory
Child module	Reusable module
Local module	Stored locally
Remote module	GitHub / Registry

Module Structure
