Practice Guide: Setting Resource Limits in Kubernetes:
Objective: Learn to apply CPU and memory limits to containers in Kubernetes pods.
Part 1: Creating a Pod with Resource Limits.
Create a file named myapp-pod.yaml.
apiVersion: v1
kind: Pod
metadata:
  labels:
    app: webapp
  name: web1
spec:
  containers:
  - image: nginx
    name: nginx
    resources:
      requests:
        memory: "64Mi"
        cpu: "250m"
      limits:
        memory: "128Mi"
        cpu: "500m"

kubectl apply -f myapp-pod.yaml
kubectl get po
kubectl describe po web1

Modify the resource limits:
Edit myapp-pod.yaml to change the resource limits and provide diiferent pod name. For example, increase the memory limit to 512Mi and the CPU limit to 2, pod name-web2.
kubectl apply -f myapp-pod.yaml				# Apply the changes
minikube addons enable metrics-server		# enable metrix server
After 2 mins
kubectl top pod web2 						# check CPU utilizations
kubectl delete -f myapp-pod.yaml			# this as part of cleanup

Demo Guide: Taints and Tolerations in Kubernetes
Learn to use taints and tolerations to control pod scheduling on Kubernetes nodes.
kubectl get nodes				# To list the nodes
kubectl taint nodes minikube size=small:NoSchedule				# Tainted for minikube
kubectl taint nodes minikube-m02 size=medium:NoSchedule			# Tainted for minikube-m02
Creating a Pod with Tolerations:
Creating tolerant-pod.yaml										# Create a tolerant-pod.yaml file
apiVersion: v1
kind: Pod
metadata:
  name: tolerant-pod 
spec:
  containers:
  - name: nginx
    image: nginx 
  tolerations:
  - key: "size"
    operator: "Equal"
    value: "medium"
    effect: "NoSchedule"

Demo : Node Scheduling
Node Affinity: Node Affinity is used to influence the scheduling of pods onto nodes based on node labels.
Label a Node: First, label a node in your cluster:
kubectl get no
kubectl label nodes minikube disktype=ssd
kunectl label nodes minikube-m02 disktype=hhd
creating a pod with Node Affinity: nginx-affinity.yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-ssd
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: disktype
            operator: In
            values:
            - ssd
  containers:
  - name: nginx
    image: nginx
kubectl apply -f nginx-affinity.yaml

Config Map:	Creating a ConfigMap from a File.
Create a separate configuration file that stores application settings instead of hardcoding them in the application code.
creating app-config.properties
database_url=example.com/db
database_user=username
database_password=password
max_connection=100
Ensure that this file contains key-value pairs in the format that your application expects
Create a configmap from file
kubectl create configmap myapp-config --from-file=app-config.properties			# This command creates a ConfigMap with the contents of app-config.properties.
kubectl get configmap -o wide					# To list the configmap
Create a pod that uses the configmap
write a pod configuration yaml(nginx-pod.yaml) that mount the Config as a volume.
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
spec:
  containers:
  - name: nginx
    image: nginx
    envFrom:
    - configMapRef:
        name: myapp-config
kunectl apply -f nginx-pod.yaml
This configuration mounts the myapp-config ConfigMap to the /etc/config directory inside the pod.
kubectl exec -it nginx-pod -- env
Cleanup:
Delete the Pod and ConfigMap
kubectl delete -f nginx-pod.yaml
kubectl delete configmap myapp-config

Secrets in Kubernetes:
Objective:
Learn to create, manage, and use Secrets in Kubernetes for storing and handling sensitive information like passwords, OAuth tokens, and SSH keys.
Secrets: Secrets in Kubernetes are objects that store sensitive data, such as passwords, tokens, and keys, which you want to keep private.
Secrets are similar to ConfigMaps but are specifically intended for confidential data.
Note: Use the base64 command-line tool to encode your data. For example, to encode the word "password", you would use:
echo -n 'password' | base64

Creating my-secret.yaml
apiVersion: v1
kind: Secret
metadata:
  name: my-secret
type: Opaque
data:
  password: cGFzc3dvcmQ= # base64 encoded value of "password" [ echo -n 'password' | base64 ]
kunectl apply -f my-secret.yaml
kubectl get secret
We can also create a Secret directly with kubectl:
kubectl create secret generic my-secret --from-literal=password='my-password'			#This command creates the same Secret as above but without writing a YAML file.
Secrets can be used in Pods as environment variables or as files in a volume. Here's an example of using a Secret as an environment variable:
creating my-pod-secret.yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-pod
spec:
  containers:
  - name: mycontainer
    image: nginx
    env:
    - name: PASSWORD
      valueFrom:
        secretKeyRef:
          name: my-secret
          key: password
kubectl apply -f my-pod-secret.yaml
kubectl get po
We can also mount Secrets as files in a Pod. This is useful for SSH keys, configuration files, etc.:

creating my-pod1-secret.yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-pod1
spec:
  containers:
  - name: my-container1
    image: nginx
    volumeMounts:
      - name: secret-volume
        mountPath: "/etc/secret"
        readOnly: true
  volumes:
  - name: secret-volume
    secret:
      secretName: my-secret

Part 4: Best Practices for Using Secrets
Avoid Storing Large Secrets: Secrets are stored in etcd and should not be used for large files such as certificates.
Restrict Access to Secrets: Use RBAC policies to restrict who can create, view, and update Secrets.
Encrypt Secrets at Rest: Configure your cluster to encrypt Secrets at rest in etcd.
